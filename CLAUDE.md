# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

AfyaCheck is an STI/HIV risk assessment system for the Kenyan market. Users answer an adaptive
questionnaire, get an ML-based risk assessment, and can find nearby health centers. An admin
dashboard lets staff manage users, questions, and usage stats.

The system is three independently-run services:

| Service | Path | Stack | Port | Purpose |
|---|---|---|---|---|
| Backend | `src/` | Spring Boot 3 (Java 21), PostgreSQL, Flyway | 8080 | Auth, questionnaire flow, sessions, admin dashboard, orchestrates the two Python services |
| Decision tree service | `python-service/` | FastAPI, scikit-learn | 8001 | Sequences the next question based on prior answers |
| ML risk prediction service | `ml-service/` | FastAPI, scikit-learn/XGBoost | 8000 | Produces the final HIV/STI risk prediction |

The UI is a React SPA (`frontend/`), bundled into the backend jar at build time. The legacy
Thymeleaf UI is fully retired (Thymeleaf renders only the transactional emails under
`templates/email/`, including the result-notification and retest-reminder emails). See
Architecture below.

## Commands

### Backend (Spring Boot, run from repo root)

```bash
./gradlew bootRun              # start the backend on :8080
./gradlew build                # compile + test + Jacoco coverage verification (see Coverage below)
./gradlew test                 # run all tests
./gradlew test --tests "com.kimanga.afyacheck.service.DecisionServiceTest"                              # single class
./gradlew test --tests "com.kimanga.afyacheck.service.DecisionServiceTest.someTestMethod"                # single method
```

`./gradlew build`/`bootJar` also builds the React SPA (`npm ci && npm run build` in `frontend/`,
see the `buildFrontend` task) and bundles `frontend/dist` into the jar's `static/` resources:
the jar is the single deployable. Pass `-PskipFrontend` for backend-only iteration without Node.
Never commit anything under `src/main/resources/static/` (git-ignored; regenerated every build).

Required env vars with **no default**, the app fails to start (not silently runs blank) without
them: `DB_PASSWORD`, `MAIL_PASSWORD`, `KEYCLOAK_BACKEND_CLIENT_SECRET`. Everything else in
`application.properties` has a sane local default (`DB_URL` defaults to port `5432`; this
environment's local Postgres runs on `5433`, so override `DB_URL` accordingly). See
`src/main/resources/application.properties.example` for the full list, and `.env` in repo root
for actual local values (git-ignored). `.env` is loaded automatically into Spring's Environment
via the `spring-dotenv` dependency (`build.gradle`), no manual `source .env` step needed before
`./gradlew bootRun`.

The two Python services **are auto-started** by the backend (`ml.service.auto.start` /
`decision.tree.service.auto.start` both `true` in `application.properties`;
`PythonServiceManager` spawns/monitors/restarts them). Set them to `false` and run the services
manually (see below) if you want to control them yourself; `PYTHON_EXECUTABLE_PATH` must point
at a real interpreter either way (defaults to a Windows path).

API docs are generated by springdoc (`/v3/api-docs`, `/swagger-ui/index.html`) and gated behind
`hasRole('ADMIN')` in `SecurityConfig`, same as `/api/admin/**`. They describe admin endpoint
shapes too, so they're not left publicly browsable.

### Python services

```bash
cd python-service && pip install -r requirements.txt && python decision_tree_service.py   # :8001
cd ml-service && pip install -r requirements.txt && python app.py                          # :8000
```

Both scripts call `uvicorn.run(..., reload=True)` when run as `__main__`, which requires an
import-string app reference and will crash on the reload check. Run them via the uvicorn CLI
instead: `uvicorn app:app --host 0.0.0.0 --port 8000` / `uvicorn decision_tree_service:app --host
0.0.0.0 --port 8001`.

Each service has a pytest contract suite (`ml-service/test_app.py`,
`python-service/test_decision_tree_service.py`) that runs against the real app + real model
artifacts; CI runs `python -m pytest` per service (deps in `requirements-dev.txt`). Run them
from inside the service directory so relative model paths resolve.

### Frontend (React + Vite, run from `frontend/`)

```bash
npm run dev           # vite-react-ssg dev server, proxies /api to http://localhost:8080
npm run build          # tsc -b && vite-react-ssg build -> frontend/dist
npm run lint            # oxlint
npm run typecheck      # tsc -b --noEmit
npm run test            # vitest run (unit); e2e/ is excluded, see test:e2e
npm run test:e2e        # playwright test; runs against the full assembled Spring Boot jar +
                        # bundled SPA + real Postgres (baseURL localhost:8080), not vite dev alone
npm run budget:check    # bundle size budget check (scripts/check-bundle-budget.mjs, budget.json)
```

`frontend/dist` is wired into the backend build (see Backend commands above): Gradle's
`buildFrontend` task builds it and `processResources` copies it into the jar. `WebConfig`
forwards `/app/**` deep links to `forward:/app-shell.html` (an empty-root variant of
`index.html` generated by `scripts/generate-app-shell.mjs`, so the client renders instead of
mis-hydrating the prerendered Landing markup), and forwards `/about`, `/faq`, `/privacy`,
`/terms`, `/learn`, and each `/learn/<slug>` topic to their individually prerendered HTML files
(`ssgOptions.includedRoutes` in `frontend/vite.config.ts`; keep `WebConfig` and
`SecurityConfig`'s permit list in sync when adding a new prerendered route).

The build also generates a PWA service worker (`vite-plugin-pwa`, configured in `vite.config.ts`)
that precaches the app shell (JS/CSS/fonts/HTML), resilient to brief connectivity drops, not full
offline questionnaire completion, since answers still have to reach the backend. API responses are
deliberately excluded from the precache/runtime cache since they're per-session data on what may
be a shared device. Icons live in `frontend/public/pwa-*.png`.

i18n covers the nav bar and the full questionnaire/results flow (`frontend/src/lib/i18n/`), a
small custom `I18nProvider`/`useTranslation` context (not react-i18next; that pulled the shared
bundle over its budget), with `en`/`sw` locale JSON files and a switcher in `NavBar`. Question
content itself is translated server-side: `Question` carries `questionTextSw`/`descriptionSw`/
`sectionTitleSw` columns, and `DecisionService.convertQuestionToMap` picks the Swahili field (with
per-field English fallback for anything untranslated) based on `LocaleContextHolder`'s locale,
set from the SPA's `Accept-Language` header (`lib/api-client.ts`). Recommendations text from the
ML service and DecisionService's rule-based fallback are still English-only.

## Architecture

### Frontend serving model (migration complete)

The React SPA is the only UI. `WebConfig.addViewControllers()` forwards `/app/**` to the
bundled `app-shell.html` (client-side render; React Router resolves deep links), and `/`,
`/about`, `/faq`, `/privacy`, `/terms`, `/learn`, and each `/learn/<slug>` topic are prerendered
static HTML from `vite-react-ssg` (`ssgOptions.includedRoutes` in `frontend/vite.config.ts`).
`/learn` is the STI/HIV education hub (`frontend/src/pages/learn/`); its topic content lives in
`topics.ts` as static data, so adding a topic means adding an entry there plus a matching
`includedRoutes` slug, `WebConfig` forward, and `SecurityConfig` permit entry. Everything under
`/app/**` is client-rendered only (auth-walled or session-personalized, no SEO value). Auth is
Authorization Code + PKCE against Keycloak from the SPA (`oidc-client-ts`); admin pages guard
themselves via `RequireAuth` inside each `*.lazy.tsx` entry.

### Backend package layout (`src/main/java/com/kimanga/afyacheck/`)

- `config/`: Spring wiring: `SecurityConfig` (identity is fully owned by Keycloak; this app is
  a stateless OAuth2 resource server that only validates JWTs; there is no local
  login/register/password-reset path anymore, `/api/admin/**` and the OpenAPI docs require
  `ROLE_ADMIN`, everything else under `/api/questionnaire`, `/api/questions`, `/api/results`,
  `/api/health-centers`, `/api/reminders`, `/api/config` is intentionally public since anonymous
  risk-screening is a product requirement), `WebConfig` (routing described above), `RateLimitFilter`
  (in-memory per-IP fixed-window limiter: 100/min on `/api/**` generally, 5/hour on
  `/api/results/notify` and `/api/reminders`), `PythonConfig` (validates Python
  executable/scripts/model files exist at startup, exposes service-availability status; does not
  itself spawn processes), `GoogleMapsConfig`, `AdminInitializer` (seeds the admin user),
  `KeycloakUserSyncFilter` (mirrors the JWT's identity/role into the local `users` row),
  `AuditingAccessDeniedHandler` (logs denied admin access to the audit log),
  `GlobalExceptionHandler`, `OpenApiConfig` (springdoc metadata).
- `controllers/`: `DashboardController`, `HealthCenterController` (Maps key + curated `/nearby`
  search with county-fallback and STI-testing filter support), `QuestionController`,
  `ResultsController` (latest/history/export/delete, plus opt-in `/notify` to email a copy of the
  result), `ReminderController` (opt-in `/api/reminders`, schedules a one-shot retest email);
  `controllers/admin/AdminController` (dashboard stats, question CRUD, health-center CRUD,
  email-based admin promotion, audit log, and `/model-ops`: assessment analytics grouped by risk
  level/model version/day plus live health of both Python services).
- `service/`: business logic. `DecisionService` orchestrates the question flow, calling
  `DecisionTreeClient` and `MLService`. `PythonServiceManager` is the actual process-lifecycle
  manager for the two Python services (spawns/monitors/auto-restarts them via `ProcessBuilder`
  when auto-start is enabled, separate from `PythonConfig`, which only validates config).
  `HealthCenterService` does the (Haversine, no PostGIS) nearest-center lookup.
  `NotificationService` sends the opt-in result email and the retest-reminder email via
  `EmailService`. `ReminderService` schedules reminders and, on a daily cron, sends and deletes
  due ones (deleted even on send failure, since a reminder row is opt-in and time-boxed, not a
  retry queue). `DataRetentionService` runs a daily cron purging sessions (and their answers and
  risk assessments) older than `retention.days`.
- `model/`: JPA entities (`User`, `Session`, `Question`, `Answer`, `RiskAssessment`, tagged with
  the `modelVersion` that produced it, see below, `HealthCenter`, `AdminAuditLog`,
  `RetestReminder`).
- `repository/`: Spring Data JPA repos, roughly one per entity. `RiskAssessmentRepository` also
  has the grouped-aggregate queries backing `/api/admin/model-ops`.
- `DTO/`: request/response shapes actually used by the live controllers/services.
- `mail/`: `EmailService` (generic Thymeleaf HTML mail sender, used for the opt-in result email
  and the retest-reminder email, both via `NotificationService`).

### Talking to the Python services

`DecisionTreeClient` and `MLService` are both plain `RestTemplate` clients. `MLService` holds
the canonical option lists (`MARITAL_STATUS_OPTIONS`, `EDUCATION_OPTIONS`,
`WEALTH_INDEX_OPTIONS`, `CONDOM_USE_OPTIONS`) used to normalize free-form input into the values
the Python model expects; actual encoding/inference happens on the Python side. Both
`PythonConfig` and `MLService` read the single `ml.service.url` property key.

### Database

Flyway migrations live in `src/main/resources/db/migration/` (`V1` through `V14` currently:
baseline schema, seed data, Keycloak ID linkage, legacy-questionnaire cleanup, admin audit log,
the `health_center` table, `risk_assessment.model_version`, Swahili question translation columns,
the `retest_reminder` table, the question-key/question-bank alignment described in Decision tree
model below, a health-center blacklist, the `age` question's minimum value, a partner-count
question label fix, and a second question-bank expansion (V14, 17 more KENPHIA-grounded
questions, see below)). `spring.jpa.hibernate.ddl-auto=validate`, so schema changes go through
new Flyway migrations, not Hibernate auto-DDL.

### Decision tree model

`python-service/train_kenphia_decision_tree.py` trains the `DecisionTreeClassifier` on real
respondent profiles from the KENPHIA 2018 household survey (`KENPHIA_CSV` env var points at a
column-subset CSV built from the raw `.dta` files; `KENPHIA_SAMPLE` optionally caps the number of
respondents used to simulate sessions, since the full ~42k-respondent run needs more memory than
a typical dev machine has). The model's 61 question keys (44 originally, expanded by V14 with 17
more KENPHIA-grounded questions covering residence, sexual debut/marriage age, circumcision,
partner age gap, needle sharing, TB/transfusion history, migration, IPV, tobacco, polygamy, and
testing frequency) must match what the DB seeds in `question`: V10 renamed the mismatched
`hiv_test` key to `hiv_tested` (the model, training data, and `DecisionService`'s risk logic all
use `hiv_tested`, so the old key's answer never reached the model) and added the 17 question keys
the model expected but the DB never seeded. Keep the DB question bank, the training script's
`get_questions_data()`, and `decision_tree_service.py`'s mirrored flow/encoding logic in sync when
any of them change — the training script and the live service duplicate the question-flow,
relevance, and answer-encoding logic rather than sharing it, so a key added to one must be added
to all three by hand (see the training script's module docstring for the current grounded-vs-
synthetic KENPHIA-column mapping).

### Model versioning

Both Python services compute a `model_version` string at startup from the loaded model artifact's
file mtime (not a hand-maintained constant, see `HIVRiskPredictor._compute_model_version()` in
`ml-service/app.py` and the equivalent in `python-service/decision_tree_service.py`), returned on
every `/predict` / `/question/next` response. The ml-service's version is persisted onto
`RiskAssessment.modelVersion` (threaded through `DecisionService` -> `QuestionController` ->
`SessionService.saveRiskAssessment`) so every stored assessment is traceable to the model that
produced it, important for a health-risk product's auditability. Rule-based fallback paths (both
Python and Java) tag themselves with a distinct `*-fallback-v1` marker rather than a real model
version.

### Data privacy: anonymous by design

The questionnaire is intentionally anonymous: `Session`/`Answer`/`RiskAssessment` have no link to
an authenticated `User`. Self-service data export/deletion (`GET`/`DELETE /api/results`, keyed by
`sessionId`, see `ResultsController` and `SessionService.deleteSessionData`) and the opt-in result
email (`POST /api/results/notify`) both follow the same access model: knowledge of the sessionId
*is* the access control, no login required, and email addresses are used once to send a message
and never persisted. Don't add a `User`-linked path for this data without an explicit product
decision, since it would change the anonymity guarantee the rest of the flow relies on.

### Coverage gate

`build.gradle` wires `jacocoTestCoverageVerification` (minimum **90% line coverage**) into the
`check` task, so `./gradlew build` fails if coverage drops below that threshold.
